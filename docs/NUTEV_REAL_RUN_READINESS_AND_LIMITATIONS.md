# NutEV Real-Run Readiness and Limitations

This document records the current methodological risk points that must remain explicit while the NutEV/NutMEV platform evolves from an evidence engine and demo platform into a fully validated real-run thesis workflow.

## Summary

The project is strong, original and methodologically promising, but it currently has two layers at different maturity levels.

The first layer is comparatively mature:

```text
search -> deduplication -> extraction -> classification -> matrices
```

This layer is well structured and already supports the generation of evidence tables and platform views.

The second layer exists architecturally, but must be treated more cautiously until it is fully integrated and validated in the main execution path:

```text
documents -> claims -> claim evaluation -> recommendation candidates -> conflicts -> protocol readiness
```

This layer has models, tests, exports, demo-data and UI/API support, but scientific claims in the thesis or articles should only state that this layer is operational for real evidence after a real run demonstrates the complete chain without placeholders or simulated metrics.

## Main fragile point

The principal fragile point is that the project currently appears to have two layers in different stages:

1. The evidence acquisition/classification layer is already relatively robust.
2. The claims/recommendation/protocol-readiness layer is newer and must be proven end-to-end in the primary pipeline.

The `master_pipeline.py` summary may include fields such as:

- `evidence_claims_total`
- `recommendation_candidates_total`
- `conflicting_evidence_total`
- `protocol_ready_total`

These fields are useful only if they are produced by the same real execution that generated the evidence corpus, claims, recommendation candidates, conflicts and readiness outputs. If a variable is filled from demo data, placeholder logic, missing defaults or manually simulated output, it must not be described as a real scientific result.

## Rule for the thesis and articles

The thesis and articles should avoid language implying that final NutEV recommendations have already been produced unless the following are true:

1. the documents were captured or registered in a real run;
2. the evidence records were deduplicated and classified;
3. claims were extracted from the documents;
4. each claim has document-level traceability;
5. recommendation candidates were generated from supporting claims;
6. conflicts were detected and preserved;
7. human review/adjudication was performed where required;
8. protocol readiness was calculated from real outputs;
9. the final exported matrices are generated by the same real execution.

Until then, the correct wording is:

```text
The platform architecture supports an auditable evidence-to-protocol workflow.
```

Not:

```text
The protocol recommendations were finalized by the platform.
```

## Demo data limitation

`demo_data.py` intentionally creates simulated data, including values such as:

- 5 records;
- 4 downloads;
- 3 claims;
- 3 recommendation candidates.

These values are for interface testing, API testing, dashboard testing, and methodological demonstration only.

They must not be presented as scientific findings, search results, review outcomes, effect estimates, guideline counts, protocol recommendations, or thesis results.

Recommended wording:

```text
Demo data were used only to verify platform behavior, output structure and user interface navigation.
```

## Anti-placeholder rule

For qualification, article submission or thesis text, avoid placeholder metrics such as:

```text
[N]
X records
Y documents
Z recommendations
```

unless they are replaced by values generated from a real, documented run.

Every reported number should have a corresponding output artifact, such as:

- `metadata_master.csv`
- `NUTEV_GLOBAL_EVIDENCE_MATRIX.xlsx`
- `NUTEV_EVIDENCE_CLAIMS.xlsx`
- `NUTEV_RECOMMENDATION_CANDIDATES.xlsx`
- `NUTEV_CONFLICTING_EVIDENCE.xlsx`
- `NUTEV_PROTOCOL_READINESS_MATRIX.xlsx`
- `run_summary.json`

## Required validation before strong claims

Before stating that the full NutEV evidence-to-protocol workflow is operational for real evidence, run and document a small real pilot, ideally with 20 to 30 documents.

The pilot should verify:

- successful source discovery;
- deduplication;
- document retrieval or metadata-only fallback;
- extracted text availability;
- classification by NutEV lenses/domains;
- claim extraction;
- claim evaluation;
- conflict detection;
- recommendation candidate generation;
- human review queue generation;
- protocol readiness generation;
- final export of all required tables and reports.

## Minimum real-run acceptance criteria

A real run can be described as a validated pilot only if it generates, from the same project output directory:

```text
02_metadata/metadata_master.csv
02_metadata/NUTEV_EVIDENCE_CLAIMS.csv
02_metadata/NUTEV_RECOMMENDATION_CANDIDATES.csv
06_tables/NUTEV_GLOBAL_EVIDENCE_MATRIX.xlsx
06_tables/NUTEV_PROTOCOL_TRANSLATION_MATRIX.xlsx
06_tables/NUTEV_EVIDENCE_CLAIMS.xlsx
06_tables/NUTEV_RECOMMENDATION_CANDIDATES.xlsx
06_tables/NUTEV_HUMAN_REVIEW_QUEUE.xlsx
06_tables/NUTEV_EVIDENCE_CONVERGENCE_MATRIX.xlsx
06_tables/NUTEV_PROTOCOL_READINESS_MATRIX.xlsx
```

If any of these are missing, the missing item should be reported as a limitation or pending integration step.

## Recommended next engineering step

The next technical priority is full integration of the scientific rigor layer into the main pipeline, not only into demo-data.

The desired pipeline contract is:

```text
run pipeline
  -> produce evidence records
  -> classify records
  -> extract claims
  -> evaluate claims
  -> detect conflicts
  -> generate recommendation candidates
  -> export audit trail
  -> export convergence/gap/readiness matrices
  -> write run_summary from real generated objects
```

The `run_summary.json` should be generated only from real objects produced during the same run. It should not silently mix real outputs and demo outputs.

## Safe language for current stage

Use:

```text
The NutEV platform provides an auditable architecture for translating evidence into recommendation candidates and protocol-readiness outputs, with human review safeguards.
```

Avoid:

```text
The NutEV platform has already generated final dietary recommendations.
```

Use:

```text
A real-run pilot is required before reporting final counts, protocol-ready items or definitive recommendation outputs.
```

Avoid:

```text
The demo-data outputs represent scientific findings.
```

## Bottom line

The project is original and strong because it does not only search for studies. It proposes a governance architecture for moving from evidence to protocol with auditability and human review.

Its main methodological vulnerability is overclaiming before a real run proves the second layer end-to-end.

Therefore, demo-data should be used for testing and demonstration; real-run outputs should be used for qualification, articles and thesis claims.
